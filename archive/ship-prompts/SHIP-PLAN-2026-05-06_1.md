## Priority dimensions

Each bundle is scored on four dimensions:

- **Visibility** — how often John encounters the bug (daily / weekly / rare)
- **Trust impact** — does this make John doubt the app's numbers? (high / medium / low)
- **Effort** — implementation cost (tiny / small / medium / large)
- **Risk** — likelihood the fix breaks something else (nil / low / medium)

The ship order maximises *visible-trust-win-per-unit-of-effort* — the
cheapest, lowest-risk wins ship first.

### 4. **Bundle 3 — Plan persistence consolidation** 🛠 dedicated mission

| Dimension | Score |
|---|---|
| Visibility | Surfaces only when goals/trips are edited (medium frequency) |
| Trust impact | Medium-high (silent state divergence — value updates one place, not another) |
| Effort | Medium (Option A) to large (Option B) |
| Risk | Medium (touches persistence) |

**Contents:** Bug C.1 + C.2 (goals + trips persistence-split fix). C.3
and C.4 deferred until repro confirms.

**Why fourth despite real impact:** the fix has migration implications
(existing `slyght_goals` and `slyght_trips` data needs to be merged
into `S` if going to single source of truth). Wants to be its own
mission spec with proper STOP gates, not a hygiene commit.

**Two paths to choose between:**

- **Option A — extend the dual-write.** Quick patch: add a generic
  `goalIdToBucketName` mapping in `confirmGoalSavings`, replicate for
  trips. ~15 lines of code. Keeps the architectural fragility
  (two stores, must keep in sync forever).

- **Option B — single source of truth.** Migrate goals and trips
  into `S` (or derive entirely from `S.savingsBuckets` + a static
  defaults table). Drop `slyght_goals` and `slyght_trips` entirely.
  Aligns with the BILLS_VM/BRAIN architecture — goals data should live
  in S so MODEL/VMs consume it canonically. Larger change with a
  one-time migration step (read existing keys → merge into S → delete
  the keys).

**My recommendation:** Defer Option A. Spec Option B as its own
mission once Bundle 1/2 are clean. The architecture work makes Option
B natural; doing Option A is a band-aid that has to be undone later.
