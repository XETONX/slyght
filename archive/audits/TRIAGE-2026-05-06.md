## Section A — HF#2 MAX PER DAY label (3 product options)

### Investigation findings

The displayed `$19.86` value at `index.html:462` (`#max-day-display`) reads
from `MODEL.safeToSpendToday`, which is `getDynamicDailyBudget()` (line 1514):

```javascript
const dailyAvailable = (bal − billsCommitted − debtsCommitted) / daysToPayday;
return Math.min(dailyAvailable, userCap);
```

So `$19.86` is genuinely **a sustainable daily rate** — the amount that, if
held to over the remaining days to payday, wouldn't run the user out of
cash. It's NOT a hard cap. The system already handles severity classification
in the context sub-line (`max-day-context` at line 463):
- `< $20` → "Tight — [bill name] hits tomorrow"
- `< $30` → "Cautious — cover your essentials first"
- `≥ $30` → "You're on track — spend wisely"

So the threshold awareness exists. The bug is purely the label word **"MAX"**
which over-promises a cap that the system doesn't enforce.

Connor's evidence: spent $91 today (4.6× the "MAX") with no in-tile violation
warning. The context sub-line said "Tight" but the headline `$19.86` with the
word "MAX" reads as a hard limit.

### Three product options (no pre-pick — your call)

#### Option A — Rename label only (minimal surface change)

| Surface | Current | Proposed |
|---|---|---|
| Tile uppercase label (`index.html:461`) | `MAX PER DAY` | `SAFE TO SPEND TODAY` |
| Number formatting (`:462`) | `$19.86` | `$19.86` (unchanged) |
| Context sub-line (`:463`) | "Tight — Adobe hits tomorrow" | (unchanged) |

- **Why "SAFE TO SPEND TODAY":** the internal field is literally named
  `safeToSpendToday` (line 2487). Aligns label with code intent.
- **Pros:** ~1 line edit. No layout change. No new logic. Honest copy.
- **Cons:** doesn't help users understand WHY today's $91 spent isn't
  flagged — the tile still doesn't communicate "you've exceeded the
  sustainable rate." Connor would still observe the $91 vs $19.86 gap
  with no warning, just the word "MAX" gone.
- **Estimated scope:** 5 lines (label + comment + maybe 1 doc reference).

#### Option B — Rename + add violation feedback when today's actual spend exceeds the rate

Same label change as A, plus tile color/copy mutation when `todaySpent > safeToSpendToday`:

| State | Tile color | Sub-line copy |
|---|---|---|
| `todaySpent ≤ safeToSpendToday` | green/normal | (existing context line) |
| `todaySpent > safeToSpendToday` | amber/red | `Spent $91 today — $71 over your sustainable rate` |

- **Why:** closes the loop on Connor's confusion — when the user has
  exceeded the rate, the tile says so explicitly instead of just dropping
  a "Tight" hint.
- **Pros:** semantic correctness. The tile becomes self-explanatory.
- **Cons:** medium scope (~25 lines). Needs `getTodaySpent()` cross-check.
  May overlap with the existing dashboard alert system (Mission E
  already deals with mode-based suppression — would need to verify
  the new state doesn't double-fire with `dash-alerts`).
- **Estimated scope:** 25-40 lines.

#### Option C — Restructure tile to show both pace and rate side-by-side

Tile becomes a two-row mini-comparison:

```
┌─ DAILY PACE ────────────────────────────┐
│  Today: $91         Sustainable: $19.86 │
│  $71 over the rate that gets you to     │
│  payday with a buffer                   │
└─────────────────────────────────────────┘
```

- **Why:** makes the relationship visible without requiring the user to
  do mental math.
- **Pros:** clearest semantic. Educational for first-time users.
- **Cons:** larger visual change (~45-60 lines incl CSS). Visual baseline
  regen required. Higher risk of layout side-effects on the surrounding
  cards. Deeper UX decision (moves the tile from headline-number to
  comparison-table).
- **Estimated scope:** 45-60 lines + CSS + visual baseline.

### Recommendation framing

- **A** is "stop lying with the word MAX." Cheap, honest.
- **B** is "make the tile self-explanatory when violated." Closes Connor's
  feedback loop directly.
- **C** is "redesign the tile around the actual mental model." Best
  long-term, biggest risk surface.

? **Decision needed** — A, B, or C (or A+B as a combined modest path).
