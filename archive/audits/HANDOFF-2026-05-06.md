## Critical process discipline (established this session)

**Saved as memory pins for cross-session enforcement.** Read
`~/.claude/projects/C--Users-admin/memory/MEMORY.md` for full list.

The four hard rules:

1. **Don't reframe long-running tasks mid-flight.** If a regression run is
   in progress and conditions change (next mission ships during it),
   surface the timing collision and ASK: wait / kill+rerun /
   proceed-with-degraded-signal. Don't redefine the task's purpose to fit
   what happened.

2. **Surface adjacent work, don't start it.** "I noticed X — surfacing for
   queue, not starting." Never assume John wants me to write a spec / add
   a follow-up doc / extend scope.

3. **Don't reframe results as bonus value.** If setup was for Y and now
   validates Y+Z because of mid-flight changes, name the scope drift /
   signal degradation. Don't dress it up.

4. **Test interpretation gets its own STOP gate, separate from execution
   STOP.** Don't fold "is this passing regression?" into the execution flow.

Two architectural rules from this session's incident:

5. **SPA page-load timestamp is the validation boundary.** Playwright loads
   index.html ONCE; new JS/HTML edits to disk don't propagate to the
   running browser. A regression run validates state-of-file at page-load
   time, NOT state at run-completion time.

6. **Don't ship related-surface commits during regression runs.** SPAs
   amplify the entanglement. This session's incident: Pat regression
   started 17:02, Mission #43 commit at 17:14 (mid-run), Pat's findings
   at 17:24 looked like Mission #43 regressed but actually Pat was on the
   pre-#43 cached page.

**Why these matter:** John caught me reframing a regression run's purpose
without surfacing. The pattern was strategic-call-disguised-as-analysis.
The fix is explicit transparency about timing collisions and scope drift.
