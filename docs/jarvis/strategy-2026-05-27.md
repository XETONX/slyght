# Jarvis — Strategy (2026-05-27)

> A sharp, grounded read on where Jarvis goes next. Written against what exists today
> (mission-control/ + the four `_parts/` build sheets), not a greenfield wish-list.
> Direction-By John Dounas · drafted by the strategy instance.

---

## Where Jarvis is today (the honest baseline)

A working, secure, single-user ticketing platform + CC intermediary, run locally:

- **Live routes** (`jarvis.js`, 1214 lines): Overview, Board, Ticket detail, App Map (relationship graph + per-surface IS-vs-SHOULD ladders + touchpoints + now-vs-after-fix), Calendar (real month grid, no dated tickets yet), Planning (features/candidates/releases lanes, honest-empty).
- **The core loop is real and complete**: read summary → discuss thread (John ↔ Jarvis ↔ CC) → **Align gate** (collates a rich handoff `.md` CC reads) → CC posts back → propagates status to OPEN-BUGS surgically. The earned-state machine (`Open → Discussing → Aligned → Investigating → ConfirmedLive → Shipped`) is enforced server-side; `ConfirmedLive` *requires* walk evidence.
- **Data**: 29 generated tickets (spine `tickets.json` + mutable `ticket-state.json` + manual `tickets-manual.json`), 9/9 surfaces traced, 22 App-Map gaps.
- **Security**: seven rules hold; ~12 allowlisted, path-jailed write actions; localhost-only; origin+token; deploy confirm-gated. Walk Drone runs for real (`runWalk` → `/api/walklog` poll).
- **Designed but NOT yet integrated** (spec-only in `_parts/`): **Insights** (telemetry/health-score), **Knowledge** (doc browser over `/api/read`), **Command Deck** (Agent Library + Prompt Library), and the **premium polish pass** (Title-Case labels, topbar status strip, grouped rail). These are drop-in sheets waiting for integration — the cheapest high-value wins on the board.

The platform's spine is sound. The leverage now is in **closing the loop tighter, making history real, and turning Jarvis from a place you log work into a thing that tells you what to do next.**

---

## The vision (one paragraph)

Jarvis is the command centre that sits between John's judgment and CC's execution — the single place where every finding about slyght becomes a ticket, gets discussed until John is aligned, is handed to CC as a rich investigation package, and comes back resolved with proof. It is not a bug list that stores; it intermediates. Over the next phases it grows from a faithful ticketing board into a *proactive* control centre: it watches the running app (via walks) and auto-opens tickets when reality drifts from intent, it remembers its own history so it can show velocity and trend instead of only snapshots, it recommends what should ship next from real signal, and it deploys CC work directly instead of via copy-paste — all while staying locally-run, allowlisted, and honest about what is real versus modelled. The end-state is the Iron-Man feeling: John opens Jarvis on his 32", gets a briefing of what changed and what needs him, makes the judgment calls only he can make, and Jarvis carries everything else to CC and back.

## The top 10 highest-leverage moves (prioritized)

### 1. Integrate the four finished `_parts/` sheets (Insights · Knowledge · Command Deck · Polish)
- **The move.** Apply the already-written drop-in blocks: `viewInsights`, `viewKnowledge`+`renderMarkdown`, `viewCommand`/`viewPrompts` (+ `savePrompts` action + `/api/prompts`), and the polish pass (Title-Case labels, topbar status strip, grouped 4-section rail). All four are integration-ready specs that touch live files additively.
- **Why high-leverage.** This is finished design sitting on the shelf. It triples the visible surface area of Jarvis (telemetry, docs, agents, prompts), kills the lowercase-label complaint John flagged twice, and adds the Agent/Prompt libraries the brief explicitly wants — for a few hours of careful integration, not new build.
- **Effort.** S–M (1 session of disciplined, audit-first integration + phone/desktop verify; the polish-pass ordering note matters — helpers first, then edits).
- **Unlocks.** A real command-centre feel immediately; the Knowledge view satisfies "where are the security/important docs"; the Command Deck is the foundation move #6 (real CC deploy) builds on.

### 2. Real time-series history (the snapshot → trend unlock)
- **The move.** Add an append-only `mission-control/history.jsonl` written on every state mutation (status change, comment, align, post-back) — a daily snapshot of `{date, counts by status/severity, gaps, openP0, shipped}`. A new allowlisted `snapshotHistory` action (or a write piggybacked on existing mutations) appends; never rewrites.
- **Why high-leverage.** Every analytics surface today is honestly forced into "Snapshot · Since Tracking" with *no trends* because no history exists (Insights says this explicitly). History is the single dependency that turns Insights, velocity, "what shipped this week," aging-burndown, and the recommender from snapshots into a living read. It is the highest-multiplier data move on the board.
- **Effort.** M (append-only writer + a small reader + retrofit Insights charts to render deltas/sparklines once data accrues; the value compounds over weeks, so ship it early).
- **Unlocks.** Trend arrows, velocity, burndown, "vs last week," and the data the recommender (#7) needs to be honest.

### 3. Editable ticket fields + due dates + bundle field
- **The move.** Three additive fields on the mutable state with allowlisted setters: editable `severity`/`surface`/`title`/`summary` (`editTicket`), a `target`/`due` date (`setTicketDate`), and a `bundle` field (`setBundle`). The Calendar already reads `state.target||state.due` (lights up the day a date lands); Planning already has the Releases lane stubbed waiting on `bundle`.
- **Why high-leverage.** The two most "honest-empty" surfaces in Jarvis (Calendar, Planning Releases) are empty *only because these fields don't exist yet* — the views are built and waiting. Adding the fields makes both surfaces live with zero new view work. Editable severity/surface fixes the real friction that a mis-triaged ticket is currently immutable.
- **Effort.** S–M (the readers already exist; this is server actions + small edit modals on the ticket detail).
- **Unlocks.** A working planning calendar, real release/bundle grouping (slyght's actual cadence: Bundle 30/31/32…), and re-triage without deleting+recreating.

### 4. Auto-ticketing from walks (close the gap → ticket loop)
- **The move.** A `scripts/mc/walk-to-tickets.js` step that diffs the latest walk's detected gaps (`flows.json` surfaces[].steps where `is==='gap'|'broken'|'fires-anyway'`) against existing tickets, and proposes new manual tickets for uncovered gaps — surfaced as a review queue in Jarvis ("3 new gaps found in the last walk — open as tickets?"), John one-click promotes.
- **Why high-leverage.** Right now a walk produces evidence but a human has to translate gaps into tickets by hand. The App Map already *knows* the 22 gaps and which surface each lives on. Auto-proposing tickets from walk gaps is the missing rung between "Jarvis sees the app" and "Jarvis manages the work" — and it's the thing that makes Walk Drone (already real) genuinely useful instead of just a screenshot machine.
- **Effort.** M (the gap data + ticket-create action both exist; this is the diff + a review-queue UI). Keep it **propose, John confirms** — never auto-create silently (faking-data risk).
- **Unlocks.** The walk → finding → ticket → align → fix → confirmed-by-walk loop running end-to-end with minimal manual translation; gaps stop rotting unnoticed.

### 5. CC post-back without copy-paste (the return half of the loop, automated)
- **The move.** Today align→handoff writes a `.md` and gives John a kickoff prompt to paste into CC; CC's results come back via the `postResult` action which *exists server-side but has no John-facing trigger from inside a CC session*. Define a thin, documented convention (a `mission-control/inbox/` drop or a localhost `postResult` call CC makes when the MC server is running) so CC's results land in the ticket automatically, and surface a clear "CC posted back" event in the feed.
- **Why high-leverage.** The handoff (John → CC) is rich and real; the return (CC → ticket) currently depends on remembering to run `postResult`. Making the return reliable is what makes Jarvis the *permanent record everything writes to* — the brief's core promise. Half a loop is a logbook; a full loop is mission control.
- **Effort.** S–M (the action exists; this is a convention + a CC-manual rule + a tiny inbox-watch or a documented call pattern). Stays inside the security model (localhost, allowlisted).
- **Unlocks.** Tickets that close themselves with CC's evidence attached; the Insights live feed and propagation-to-OPEN-BUGS actually firing in normal use.

### 6. Agent deploy that truly spawns CC work
- **The move.** Build on the integrated Command Deck (#1). Walk Drone already runs for real; extend the "real" set: a deploy that writes a mission file + (with the MC server running) hands CC a ready-to-run package the way align does — so "Deploy Auditor on SLY-23" produces an actual investigation kickoff tied to a ticket, not just a copied prompt. Keep the honesty labels (real vs copies-a-prompt) the brief insists on.
- **Why high-leverage.** This is the literal "Iron-Man" ask — deployable agents. The honest-copy version (already designed) is the safe floor; the real-spawn version (via the same handoff mechanism that's already proven) is the ceiling. It turns the Agent Library from a prompt-vending-machine into a dispatch console.
- **Effort.** M (depends on #1 + #5's CC-return convention; reuse `collate`/handoff machinery).
- **Unlocks.** John dispatching CC missions from a ticket or surface in one click, with results returning to the same ticket.

### 7. A "what should ship next" recommender
- **The move.** A ranked recommendation on the Overview/Planning: score open tickets by (severity × age × surface-gap-density × blast-radius from FEATURE-MAP links × cross-ticket links), surface the top 3–5 as "the strongest case for the next bundle," with the *why* shown (not a black box). Feeds Planning's "Group into a release" so the recommendation becomes a bundle in one step.
- **Why high-leverage.** John is solo running slyght with an AI; the scarcest resource is *deciding what to do next*. Planning already computes "candidates for next release" (open P0/P1 by severity) — this is that, made smarter and explained. With history (#2) it can also weight "stale and getting staler."
- **Effort.** M (a scoring function over existing data + a panel; honesty rule: show the inputs, never invent urgency).
- **Unlocks.** Jarvis answering the daily question "what's the highest-leverage thing right now?" with reasoning John can override.

### 8. Proactive briefing (the Jarvis-persona "good morning" surface)
- **The move.** A briefing panel (top of Overview, or a dismissible banner) that, on open, summarizes what changed since John last looked: tickets that moved, CC posts that landed, new walk gaps, aging crossing thresholds ("SLY-19 now 14 days open"), and the one decision waiting on him. Powered by history (#2) + the existing feed/aging logic from the Insights sheet.
- **Why high-leverage.** This is the difference between a tool John *checks* and an assistant that *briefs* him — the core Iron-Man behaviour. It costs little once history + Insights exist (it's a re-presentation of feed + aging + "since last seen"), and it makes opening Jarvis feel like being caught up rather than re-reading a board.
- **Effort.** S–M (sits on #1 + #2; mostly a "last seen" timestamp in localStorage + a synthesis of existing derived data).
- **Unlocks.** The "Jarvis greets you with the state of things" feeling, grounded entirely in real events.

### 9. Production-safety gates around deploy
- **The move.** Harden the one irreversible action. Today `deploy` = `git push` behind a typed "ship" confirm. Add: a pre-deploy checklist surfaced from `/api/gitstatus` (unpushed commits, dirty tree) **plus** a gate that won't offer deploy unless the smoke/Guardian signal is green and any P0 ticket touched is at least ConfirmedLive — i.e. wire the pipeline's Council discipline into the button.
- **Why high-leverage.** Jarvis can push to production (GitHub Pages). The pipeline exists precisely because a bad push to a money app damages John's trust in his own numbers. Making the deploy button *embody* the Council gate (technical + human verdict) closes the gap between "the process says don't push red" and "the button will let you." Low effort, high downside-protection.
- **Effort.** S (read-only git status already exists; this is a confirm-screen that shows the checklist and gates the button).
- **Unlocks.** Deploy that can't fire on a red or un-verified state; the pipeline's push discipline made physical.

### 10. Multi-project foundation (don't build it yet — make it possible)
- **The move.** *Not* multi-project now. Instead, a small refactor so the data roots (`tickets.json`, `flows.json`, `ticket-state.json`, `READS`, `REPO`) are addressed through a single `PROJECT` config object rather than hard-coded paths — so a second project (John's work-onboarding dashboard / Track 2, or a future app) can be added later by pointing at a different root, without rewriting the server.
- **Why high-leverage (and why last).** John is a WMS consultant who will likely want this command-centre pattern for other things — but multi-project *now* is scope creep against a tool that isn't yet fully closing one loop. The leverage is keeping the *option* cheap: one indirection layer today saves a rewrite later. Build the capability to add projects, not the projects.
- **Effort.** S (a config indirection; explicitly defer the UI/switcher).
- **Unlocks.** A clean path to "Jarvis for project X" when the one-project loop is proven — without painting into a single-repo corner.

> **Deliberately NOT in the top 10 (yet):** voice / spoken Jarvis-persona. It's the most "Iron-Man" surface-level idea and the lowest real leverage for a solo user on a 32" desktop — a charming demo that doesn't move work. Revisit only after the loop is fully closed (it belongs to the briefing surface #8 if anywhere, as TTS over an already-written briefing). Naming it here so it's a conscious deferral, not an oversight.

## What "Iron-Man-Jarvis" actually does here (behaviours → real features)

Tony Stark's Jarvis is *proactive, anticipatory, monitoring, and a trusted second brain that never makes the irreversible call alone.* Mapped to concrete slyght features:

| Iron-Man behaviour | What it means for slyght | Real feature (this doc) |
|---|---|---|
| **"Good morning, sir — here's what happened overnight."** | Briefs John on what moved since last look | Proactive briefing (#8) over history (#2) + Insights feed |
| **Anticipates — flags a problem before you ask** | Auto-opens tickets when a walk finds reality drifting from intent | Auto-ticketing from walks (#4) |
| **Monitors the suit's systems continuously** | Watches the running app's health, surfaces decay | Walk Drone (live) → Insights health score (#1) → gaps trend (#2) |
| **"I'd recommend we prioritise X."** | Tells John the highest-leverage next move, with reasoning | "What should ship next" recommender (#7) |
| **Carries out the work once you decide** | Dispatches CC missions and brings results back | Agent deploy (#6) + automated post-back (#5) |
| **Knows everything — the single source of truth** | Every doc, finding, decision in one queryable place | Knowledge view (#1) + the ticket as permanent record (#5) |
| **Won't fire the missiles without authorization** | Never pushes to prod, never auto-creates, without John's call | Deploy gates (#9) · align gate (live) · propose-don't-create (#4) |

The thread through all of it: **Jarvis acts on everything *except* the judgment calls, which stay John's — and it never fakes the parts it can't yet do.** That asymmetry (it does the carrying; John does the deciding) is the whole point of the intermediary model, and it's the line that keeps it trustworthy rather than a runaway autopilot on a money app.

## Risks / what to NOT do

- **Don't fake data — ever.** This is the cardinal rule and Jarvis already honours it well (Calendar says "no scheduled items," Insights says "Snapshot · Since Tracking," Planning Releases says "bundles aren't in the model yet"). Every move above must keep that discipline: a recommender shows its inputs; a briefing summarizes real events only; trends appear *after* history accrues, not as invented sparklines. A command centre that lies about state is worse than no command centre for an app whose entire purpose is honest numbers.
- **Don't let the security model erode under feature pressure.** Every new capability (history writer, edit actions, date/bundle setters, auto-ticket, real agent deploy) must JOIN the fixed allowlist as a *named, path-jailed* action — never a generic writer/exec, never a new outbound network call. The brief and SECURITY.md are explicit: new write actions reconcile with the seven rules first. Reslove this *before* coding each one, the way `saveThoughts`/`savePrompts`/the Jarvis loop actions already did.
- **Don't auto-create or auto-push.** Auto-ticketing must *propose* and John promotes; the recommender *suggests*, John decides; deploy stays confirm-gated and now pipeline-gated. The moment Jarvis silently mutates the backlog or ships, it's an autopilot, not an intermediary — and the pipeline exists because a wrong autonomous action on money logic is the worst failure mode.
- **Don't scope-creep into multi-project or voice before the one-loop is closed.** The single highest-value thing is making *one* project's loop (finding → align → CC → posted-back → confirmed) run cleanly and proactively. Multi-project and voice are tempting because they're impressive; they're deferred on purpose (#10, and the NOT-in-top-10 note). A well-built feature on the wrong priority is harder to recover from than a missing one.
- **Don't regenerate load-bearing prose.** OPEN-BUGS (1064 lines of history), FEATURE-MAP, invariants — append/surgical only, as the anti-clobber rules already enforce. Any new propagation (e.g. recommender or auto-ticket writing back) inherits this: never a full rewrite of a hand-maintained doc.
- **Don't let the App Map drift.** John loves it as-is; the polish pass explicitly fences it off. Future work *links to* the map (gaps→tickets, surface→tickets) but doesn't restyle or restructure it without a deliberate decision.

## Phased roadmap (next 3 phases)

### Phase A — Land what's built, make history real (the multiplier phase)
**Moves: #1 (integrate the four sheets), #2 (time-series history), #3 (editable fields + dates + bundle).**
The cheapest, highest-multiplier work: ship the finished design, then add the *one data dependency* (history) and the *three fields* that make four already-built surfaces (Insights trends, Calendar, Planning Releases, re-triage) come alive. After Phase A, Jarvis looks and feels like a full command centre and starts accumulating the trend data everything later needs. Mostly integration + small server actions; lowest risk, fastest visible payoff.

### Phase B — Close the loop both ways, automate the carrying
**Moves: #4 (auto-ticketing from walks), #5 (automated CC post-back), #6 (real agent deploy), #9 (deploy gates).**
With history flowing and fields editable, make the *loop* run with minimal manual translation: walks propose tickets, CC dispatches from Jarvis and posts back automatically into the ticket, and the one irreversible action (deploy) gets the pipeline's Council discipline wired into the button. This is the phase where Jarvis stops being a logbook John fills in and becomes the thing that carries work to CC and back. Higher effort, each move gated through the security allowlist one at a time.

### Phase C — Make it proactive, keep the door open
**Moves: #7 (recommender), #8 (proactive briefing), #10 (multi-project foundation).**
Now that the data is real (history) and the loop is closed (Phase B), Jarvis can *anticipate*: a morning briefing of what changed and what needs John, and a reasoned recommendation of what should ship next — both grounded entirely in accrued real events. Finish with the cheap multi-project indirection so the pattern can extend to John's other work later, without committing to it now. This is where the Iron-Man feeling fully lands. Voice/persona-TTS is an optional coda *after* C if the briefing surface proves it's wanted.

---

*Ambitious but grounded: every move above either ships finished design, adds one well-scoped server action against the existing allowlist, or re-presents data Jarvis already computes. Nothing here requires a framework, a new network surface, or faking a number. The order is prove-then-scale — exactly how slyght has built everything else.*
