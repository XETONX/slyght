### 2.4 localStorage write sites

There are TWO storage layers:

1. **`STORAGE_KEY = 'slyght_v5'`** — the main `{S, BILLS}` blob. Written only by `save()` at line 1180.
2. **Side-channel keys** — written directly by various consumers:
   - `slyght_api_key` — the Anthropic key, kept separate so exports can strip the main blob without losing it
   - `slyght_audit_log` — `AUDITOR.record` line 6839 writes a 50-entry slice
   - `slyght_payday_plan` — Plan Mode lock state (line 10458)
   - `slyght_last_cat` — last quick-log category (UX continuity)
   - `slyght_onboarded` — flag preventing re-onboard
   - `slyght_seeded_v9..v15` — migration flags
   - `slyght_last_export` — APP_HEALTH EOM reminder
   - `slyght_chat_inactivity_*` (likely)

Side-channel keys are **outside the `S` save/load round-trip** — they don't appear in `state-snapshot.json` and aren't restored when state is restored from a snapshot. **Layer 1 candidate rule:** every `localStorage.setItem` call site should be cataloged and either go through `save()` (and thus through the main blob) or be explicitly justified as a side-channel.

### 10.1 The pipeline

       ↓
GitHub Pages action (.github/workflows/pages.yml)
       ↓
Repo uploaded as Pages artifact (no build step)
       ↓

       ↓

       ↓

### 10.2 Timing

- Push to live: **~30–90 seconds** typical for Pages
- Service worker swap: on next page load (or `sw.skipWaiting()` immediately for new clients)
- No staging environment — every push is production

### 10.3 What runs server-side

**Nothing for the app itself.** GitHub Pages is static. The only server is the Cloudflare Worker (`slyght-worker/`) which schedules push notifications. Worker deploys via `wrangler deploy` separately — not coupled to repo push.

### 10.4 Pre-commit / pre-push hooks

**None.** `git config core.hooksPath` not set. Tests/guardians are run manually (or, in our missions, by Claude Code before pushing).

### 10.5 Where Layer 3 (Opus audit) could plug in

Three options:

1. **Pre-push hook locally** — runs `node guardian-all.js && node tests/core.test.js && opus-audit` before push. Reversible (uninstall hook).
2. **GitHub Actions step** — add a `verify` job before `deploy` in `pages.yml`. If verify fails, deploy blocks. Requires Anthropic API key in repo secrets.
3. **Mid-mission gate** — Claude Code (this session) calls Opus audit as a sub-agent before declaring a mission shipped. Already happens informally; could be formal.

Option 2 is the strongest gate but adds a paid API call to every push. Option 1 is free but bypassable. Option 3 is hardest to enforce but cheapest.

| `check*` | Trigger + maybe action | `checkForBonusInterception`, `checkForWrxSaleInterception`, `checkChatInactivity`, `checkDayBudgetAlert` |

### 13.3 Things I noticed that aren't bugs but Guardian should know

- The codebase grows new "engines" (capitalized constants like `AUDITOR`, `NOTIFY`, etc.) at a rate of roughly one per major mission. There are 22 named engines. Guardian should be able to enumerate them dynamically rather than hard-coded.
- Most engines are objects with `init()`-like patterns called from window.onload. If Guardian needs a startup-correctness check, the engine init order matters. (TDZ bugs already happened — `c800400` and `56896d8` both addressed boot-order issues.)
- Migration flags like `_paidBillsKeyMigrationV1` accumulate — currently 4–5 of them in `S`. Guardian should know these are write-once-then-read-only and shouldn't surface as inconsistencies if missing.
- The seed function (`REAL_STATE_V17` at line 6663) is the largest single block of fixture data. Any change to schema needs to be reflected in the seed AND in `state-snapshot.json` AND in `tests/core.test.js`'s inline mock — three places that drift.
