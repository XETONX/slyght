## Executive summary (1 page)

### Top 5 findings (the things that showed up under multiple lenses)

1. **Data-shape fragility — same entity exists in 2-4 places without canonicalisation.** Engineer, Design Lead, and Financial Advisor independently flagged this. China lives 3× (bucket / trip / goal); NRMA / Teachers Health / WRX Rego / KIA Green Slip span 2-4 surfaces each. The codebase can't tell you "what is China." Bundle 28 §14.0's `PLAN.intents` is the right fix; it's the **single highest-leverage debt in the project**.

2. **Cross-tile metric divergence — three tiles, three numbers, all claiming to be authoritative.** Footer "$22 today", Dashboard "Over by $44", Analysis "$74.09" (OPEN-BUGS #17). Same for daily-cost figures ($38.65 / $41.56 / $16.56). Bundle 19's `BRAIN.summary` foundation is laid but not consumed by the divergent renderers. Engineer + CI Leader + Design Lead all named this independently.

3. **Stale "Phase N candidate" / "coming soon" stubs corrode user trust.** Bundle 27's "Ask AI" toast still says "coming Phase 7" though Phase 7 wiring landed. KIA Extra editor is `alert('Phase 6 candidate')`. Process gap: stub-strings have no shelf-life enforcement. CI Leader's recommendation: Layer 1 rule `no-stale-phase-message` (already in Bundle 28 spec §17.1.1).

4. **Doc decay — 60+ MD files in slyght/, ~10 superseded, no retention policy.** PM, CI Leader, and the cleanup sweep all flagged this. Any new contributor (Claude or human) wades through 36 MISSION-*.md to find the canonical roadmap. ARCHITECTURE.md is the load-bearing index; many MISSION docs are historical artifacts.

5. **The financial logic is sound at the cycle level but thin at the long-horizon level.** Financial Advisor's verdict: slyght is excellent at "payday-to-payday optimization" but loses cohesion at quarter/year/lifetime. No retirement-readiness modelling, no tax visibility, no debt-payoff-race tool that would show John his $25K WRX-sale proceeds could clear all non-rent debts in 90 days if sequenced right.

### Top 5 recommendations (in priority order)

1. **Ship Bundle 28 phase 0 (PLAN.intents) before any further UI work.** Every downstream surface reads through the canonical entity. Without it, Bundle 28's UX phases would re-encode the same fragility in new code.
2. **Add Layer 1 static rule `cross-tile-metric-canonical`** — bans inline `S.txns.filter(...)` outside the canonical-helpers set. Forces renderers to consume `MODEL.todaySpent` / `BRAIN.summary.total`. This is the structural answer to finding #2.
3. **Cleanup pass on slyght/ folder** — proposed structure in §"Working directory + repo cleanup" below. Move ~25 docs to `slyght/archive/missions/`, keep working artifacts at root.
4. **Add a "Retirement Readiness Canvas" + "Debt Payoff Race" tool to PLAN mode** — these unlock genuine financial decisions. Financial Advisor flagged them as the highest-leverage features for John specifically given his situation.
5. **Adopt a quarterly audit cadence** — this audit becomes the template. Run again at Bundle 32 (~10 bundles forward), or earlier if triggered by a revert / a "horrid" rejection / a discovered data-shape duplication.

### The strategic answer

**Yes, slyght is helping you.** Bundles 1-27 have built a measurably useful daily-spend tool with a guarded canonical writer pattern that prevents regression. But the *trajectory* has drifted: more engineering, less explicit measurement of whether John's actual finances are improving from using the app. See Lens E "the chat trap" — it's not a chat trap yet, but it's becoming one if Bundle 28+ keeps shipping engineering polish without an explicit "did I save more / pay debt faster / spend less" KPI on John's side.

## Lens A — Engineer

*"If a senior engineer joined the project tomorrow, what would they find and what would they say to clean up?"*

### A1. Architecture map

slyght is a single-file mobile-first PWA (~14,500 LOC `index.html`) with localStorage persistence and a service worker. Architecture is **canonical-writer-with-audit-log** centered on 11 BRAIN bubbles (audit · config · savings · summary · dashboard · transaction · bills · debts · allocation · plan · pending assets). Each bubble owns its slice of state; every write validates a frozen SOURCES tag and appends to `BRAIN.audit`. Calc layer (12 pure helpers) computes derived numbers per render; MODEL is rebuilt on every `renderAll()`. Render layer is innerHTML rip-replace (no targeted DOM updates). Three-layer guard system (Layer 1 static AST · Layer 2 runtime synthetic-state tests · Layer 3 MathInvariants per render) + new Bundle 27 boot self-test.

## Lens B — Design Lead

*"Does the app tell a coherent story across surfaces? Where does the user's mental model break?"*

### B1. Mental-model breaks (user-facing)

1. **China appears as three rows in three places** — savingsBucket "China Holiday" $61.82 / Trip "china-2026" $5000 budget / Goal "china" $5000 target. User can't tell if topping one up tops the others. Same for NRMA, Teachers Health, WRX Rego, KIA Green Slip.
2. **AnnualProvisions still names "KIA registration" / "KIA green slip" / "KIA insurance"** — but seed v17 lists "WRX Rego paid" and "WRX Green Slip paid" as historical debts. The naming drift from WRX→KIA hasn't fully landed; the user sees both names for analogous bills.
3. **"Go to Bills" routes to PLAN-mode Darwin Trips** — root cause traced in Bundle 28 §18: `closePaydayPlan(); goPage('pg-cal')` doesn't close PLAN mode itself, so user sees PLAN root over Bills tab.
4. **"Spent today" shows three different numbers** across footer / Dashboard / Analysis (OPEN-BUGS #17).
5. **"Survival" / "Tight" / "Cautious" mode labels** can disagree with the spending-cap card displayed beneath them (Mission E suppressed surface display, root math still divergent).

### B2. Vocabulary inconsistencies

- **"Paid" vs "spent" vs "used"** — Bills tab says "paid," Analysis says "spent," some debts say "cleared," "paid off," or "settled." No single verb-of-record.
- **"Discretionary" vs "non-essential"** — Layer 1 rule names it `_NON_SPEND_CATS`; UI sometimes shows "discretionary," sometimes "fun money."
- **"Surplus" vs "free" vs "available" vs "remaining"** — all four appear in PLAN-mode + Dashboard for similar concepts.
- **"Provision" vs "sinking fund"** — both terms used; user-facing copy in Savings sub-screen calls them "sinking funds for yearly costs" but the tile is literally named "Annual Provisions."

### B3. IA assessment per surface

| Surface | Order | Coherence |
|---|---|---|
| PLAN root | NW header → WRX → Payday Plan → Trips → Goals → Super → AnnualProvisions → IncomeSim | **Inverted** — should be cycle-first, long-term-second (Bundle 28 §3) |
| Settings root (Bundle 22 v3) | Samsung-style sub-screens | **Strong** — biggest IA win in the app |
| Payday Plan canvas | Essentials / Remainder / Allocations + 5 sub-screens | **OK** but Bills sub-screen mixes paid+unpaid, Savings sub-screen splits Goals/Trips harshly |
| Dashboard | Hero balance → Survival banner → Today's spend → Bills due → Debts → Recent txns | OK but tile redundancy (3 "today's spend" mentions, "$110 due this week" duplicated) |
| Bills tab | This Week → Monthly Bills | Strong but Calendar markers don't show ad-hoc debts |
| Snapshots | Flat list, ~250 entries | Cluttered — virtualization candidate (Bundle 22.5+) |
| Analysis | ~12 tiles | **Crowded** — Phase D recommends pruning to 4-5 |

### B4. Static surfaces that should be tappable

Per Bundle 28 §19 P1: *every number should be tappable to a "what's this made of?" modal.*

Currently static: "$X liquid net worth" (taps Net Worth modal — good), "$Y free this cycle" (no tap), "$Z saved toward China" (no tap), "N days until payday" (no tap), "On track / Behind pace" (no tap). All five should reveal their input math.

### B5. Highest-friction user flows

1. **"Did I already pay this bill?"** — currently requires: open Bills tab, find the bill, check the paid indicator. Should be: tap the bill anywhere it appears, see paid/unpaid + history inline.
2. **"How am I tracking on China?"** — requires opening Savings sub-screen + scrolling to bucket + mentally computing pace. Should be: one tap on China anywhere, see saved/target + projected completion date.
3. **"Can I afford this $200 purchase?"** — requires manually adding to a mental model. AI chat affordability exists post-Bundle-27 but isn't surfaced as a one-tap action.
4. **"What's eating my budget this month?"** — Analysis tab is the place, but 12 tiles + cross-tile divergence means user can't quickly answer.
5. **"How long until I'm debt-free?"** — payoff projections are scattered across Debts tab + WRX card + PLAN mode without unified timeline.

### B6. Touch-target violations

Spot-checked: `.tappable` utility enforces 44×44 universally. `.icon-btn` (34×34 circle) in Dashboard header is **borderline** — recommend bumping to 44×44 in Bundle 28+. Plan-mode hero buttons OK. Settings sub-screen back button enforces 44×44 via `.subscreen-back` class.

### B7. Design lead's verdict — top 5 UX cleanups John will feel biggest

1. **Visualise allocation in Savings sub-screen** — Bundle 28 §5 progress bars + ghost-segment-on-edit (high-impact).
2. **Collapse duplicate "China" into one canonical entity** — Bundle 28 §14.0 PLAN.intents (removes the cognitive WTF).
3. **Unify "spent today" to a single MODEL.todaySpent surface across footer / dashboard / analysis** — kills the "which number is real" friction.
4. **Tile-order swap (Payday Plan above WRX in PLAN root)** — small change, high salience.
5. **Replace stale stubs (Ask-AI toast, KIA editor alert)** — trust-restoring.

## Lens E — Strategic Leader

*"Is this app on track to serve its purpose? Should the direction change?"*

### E1. Original vision recap

slyght was scoped as a multi-domain personal accountability tool: **finance tracking + lifestyle accountability + addiction recovery + City2Surf training integration.** Current state has narrowed almost exclusively to finance.

### E2. Goal-by-goal: shipped vs deferred vs drifted

| Original goal | Status |
|---|---|
| Finance tracking | **Shipped robustly** — 27 bundles deep |
| Lifestyle accountability (no-spend streaks, etc.) | **Drifted** — Character / SLYGHT Score / 90-day forecast features cut in Phase E |
| Addiction recovery | **Drifted** — vape detection / weed detection cut Bundle 14 |
| City2Surf training integration | **Drifted** — no fitness surface |

### E3. Feature classification

- **Tracking (passive):** Quick Log, bills, debts, balances, snapshots, audit log — ~70% of surfaces
- **Coaching (active):** Survival banner, Pace tile, payday allocation suggestions, AI chat affordability — ~25%
- **Accountability (intervention):** None currently — no notifications driving behaviour, no nudges, no "you said you'd do X by Y" surface

### E4. Scope drift surface

Features that exist mostly to manage code complexity, not to help John make money decisions:
- Math Health diagnostics panel
- Activity Log (Diagnostics)
- 51 runtime guardian checks
- 13 MathInvariants
- Layer V screenshot harness
- Snapshot eviction algorithm

All are **enablers** for the engineer experience, not the user experience. They're necessary for the architecture to scale, but they're not directly serving John's wallet.

### E5. Scope shrink surface

Goals that quietly dropped off:
- Habit-streak coaching (cut Phase E)
- Smart push notifications (Bundle 25 deferred → Bundle 30+; barely working per BACKLOG #43)
- City2Surf integration (no mention since early scoping docs)
- Addiction recovery tracking (cut Bundle 14)

### E6. The chat trap question

**Honest answer:** slyght is not yet a chat trap — but it has the trajectory to become one. Evidence FOR (it's becoming one):
- 27 bundles in 6 months. That's significant engineering time.
- No KPI on John's side (no measurable "I saved $X more / paid debt $Y faster / made better decision Z because of slyght").
- Recent bundles trending toward refactor/architecture rather than user-money-impact features.
- The financial advisor lens found: zero retirement modelling, no debt-payoff-race tool, no tax visibility — features that *directly* affect John's money.

Evidence AGAINST (it's still real):
- Phone-walk discipline confirms John uses it daily.
- Bundle 22 v3 + 27 added measurable UX value (Samsung Settings, Payday Plan canvas).
- Memory + ARCHITECTURE.md show John reasoning about the app as a tool, not a project.

**Verdict: tipping point.** Bundle 28's UX-debt-cleanup is on the right side. If Bundle 29+ keeps doing architecture-for-architecture, it tips. If Bundle 29+ ships a Retirement Readiness canvas + Debt Payoff Race tool + Tax-visibility surface, it stays a tool.

### E7. Complexity-to-purpose ratio

**Current ratio: HIGH.** 14,500 LOC + 11 bubbles + 50+ source tags + 51 runtime checks + 13 invariants + 16 static rules + 50 Layer 2 tests + Layer V harness — all serving ~ a dozen user-facing surfaces. Architecture overhead is justified if Bundle 29+ leverages it (AI agent writes, multi-device sync, learned prediction). It's NOT justified if the next 10 bundles are more architecture without user-money-impact features.

### E8. Leader's verdict — direction change?

**Yes, mildly.** Current direction (architecture-first, polish-next) has served well through Bundle 27. From here:
